Spark 2.0 + WebXR 연구 프로젝트 개발 기록 (초보자용)

작성일: 2026년 7월 15일 (v2 — 디버깅 기록 추가) · 프로젝트 위치: ~/Desktop/spark-project

1. 이 프로젝트는 무엇인가요?

이 프로젝트는 가우시안 스플랫(Gaussian Splat)으로 만든 3D 공간을 웹 브라우저와 VR 헤드셋에서 탐험할 수 있게 해주는 연구용 템플릿입니다. 단순히 3D 장면 하나를 화면에 띄우는 것이 목표가 아니라, 앞으로 아바타, 월드 공간 UI, 멀티플레이어, AI 에이전트까지 얹을 수 있는 확장 가능한 뼈대를 만드는 것이 목표입니다.

현재 상태: 데스크톱 브라우저에서 가우시안 스플랫(나비 샘플)과 일반 GLB 3D 모델(로아레 성)이 한 장면에 함께 렌더링되고, 마우스로 둘러보고 WASD로 이동할 수 있으며, VR 진입 버튼이 준비되어 있습니다. 연구 질문 "스플랫과 일반 메시가 공존할 수 있는가?"에 대한 답은 "예"임을 눈으로 확인했습니다.

2. 먼저 알아야 할 핵심 개념 세 가지

가우시안 스플래팅 (Gaussian Splatting)

전통적인 3D 그래픽은 삼각형(폴리곤)으로 물체를 만듭니다. 반면 가우시안 스플래팅은 수십만~수백만 개의 작고 반투명한 얼룩(스플랫)을 공간에 뿌려서 장면을 표현합니다. 실제 장소나 물체를 여러 각도에서 촬영한 사진들로부터 만들어지기 때문에, 사진처럼 사실적인 공간을 웹에서 보여줄 수 있습니다.

Spark

Spark는 가우시안 스플랫을 Three.js(웹 3D 라이브러리) 안에서 그려주는 렌더러(그리는 도구)입니다. 여기서 아주 중요한 사실 하나: Spark는 스플랫을 "만들어주는" 도구가 아닙니다. 이미 만들어진 스플랫 파일(.ply, .spz, .splat 등)을 불러와서 화면에 그려줄 뿐입니다. OBJ나 GLB 같은 일반 3D 모델을 스플랫으로 변환할 수는 없고, 그런 모델들은 Three.js의 일반 메시(mesh)로 취급해서 스플랫과 같은 장면 안에 나란히 배치합니다.

WebXR

WebXR은 웹 브라우저에서 VR/AR 기기를 사용할 수 있게 해주는 표준 기술입니다. 앱을 따로 설치하지 않아도, Quest 헤드셋의 브라우저로 주소만 열면 VR로 들어갈 수 있습니다. 화면 하단의 "Enter VR" 버튼이 이 역할을 합니다. (VR 기기가 없는 일반 데스크톱에서는 "VR NOT SUPPORTED"라고 표시되는데, 이는 정상입니다.)

보조 도구로는 Three.js(웹에서 3D를 그리는 표준 라이브러리, Spark가 이 위에서 동작)와 Vite(개발 서버를 띄우고 코드를 묶어주는 빌드 도구)를 사용합니다.

3. 개발 전에 겪었던 문제들 (그리고 배운 점)

문제 1 — 버전 충돌: 처음 npm install @sparkjsdev/spark를 실행했을 때 "ERESOLVE" 오류가 났습니다. 프로젝트의 Three.js가 0.167 버전이었는데, Spark 2.1은 Three.js 0.180 이상을 요구하기 때문입니다. npm install three@latest로 Three.js를 먼저 올린 뒤 Spark를 설치해서 해결했습니다.

문제 2 — 엉뚱한 폴더에 설치: npm 명령을 홈 디렉터리(~)에서 실행하는 바람에, 라이브러리가 프로젝트가 아닌 홈 폴더에 설치되어 있었습니다. 실제로 프로젝트 폴더를 확인해 보니 three도 spark도 설치되어 있지 않아서, 프로젝트 안에 다시 설치했습니다(three 0.185.1 + spark 2.1.0). 교훈: npm 명령을 실행하기 전에 항상 pwd로 현재 위치를 확인하세요.

문제 3 — Spark 저장소와 내 프로젝트 혼동: Spark의 GitHub 저장소를 통째로 클론하면 build:wasm 같은 스크립트가 보이는데, 이것은 Spark 라이브러리 자체를 개발하는 사람들을 위한 것입니다. 우리는 Spark를 npm 패키지로 설치해서 쓰는 사용자이므로, 내 Vite 프로젝트에는 그런 스크립트가 없는 것이 정상입니다.

4. 1단계: 모듈형 뼈대 만들기

처음에는 모든 코드가 파일 하나(SparkTest.js)에 들어 있었습니다. 동작은 하지만, 기능이 늘어날수록 관리가 불가능해집니다. 그래서 역할별로 파일을 나누는 모듈 구조로 처음부터 다시 만들었습니다.

파일역할
src/main.js입구. 설정(CONFIG)을 정의하고 아래 모듈들을 조립한 뒤 렌더링 루프를 돌립니다.
src/scene.js렌더러·장면·카메라·조명·SparkRenderer 생성. 창 크기 변경 처리.
src/splat.js가우시안 스플랫 파일을 불러오는 함수.
src/player.js플레이어 이동: WASD 걷기 + 마우스 시점 회전(포인터 락).
src/models.jsGLB/GLTF 일반 3D 모델 로딩 (2단계에서 추가).
src/xr.js"Enter VR" 버튼과 XR 세션 시작/종료 처리.
src/ui.js화면 좌측 상단의 HUD(상태 표시 + 조작법 안내).
src/avatar.js아바타 로딩 함수. 미리 만들어 두었지만 아직 연결하지 않음(7단계용).

설계 원칙: 각 모듈은 서로를 직접 참조하지 않습니다. main.js가 모든 것을 만들고 필요한 것을 명시적으로 건네줍니다. 이렇게 하면 나중에 network.js(멀티플레이어)나 agent.js(AI 에이전트)를 추가할 때 기존 코드를 건드리지 않아도 됩니다.

플레이어 리그(rig) 패턴 — 이 프로젝트에서 가장 중요한 설계

카메라를 장면에 바로 넣지 않고, "리그"라는 보이지 않는 상자(THREE.Group) 안에 카메라를 넣습니다. 데스크톱에서는 WASD가 리그를 움직이고, 마우스가 리그 안의 카메라를 회전시킵니다. VR에서는 헤드셋이 리그를 기준으로 카메라를 움직입니다. 나중에 컨트롤러 스틱 이동이나 멀티플레이어를 만들 때는 리그만 옮기면 됩니다.

초보자가 놓치기 쉬운 중요 설정들

SparkRenderer는 장면에 반드시 추가해야 합니다. new SparkRenderer({ renderer })를 만들어 scene.add() 하지 않으면 스플랫 정렬이 제대로 동작하지 않습니다. (기존 실험 코드에는 이 부분이 빠져 있었습니다.)

안티앨리어싱은 꺼야 합니다. new THREE.WebGLRenderer({ antialias: false }) — 공식 문서에 따르면 안티앨리어싱은 스플랫 화질을 개선하지 못하면서 성능만 크게 떨어뜨립니다.

메시는 조명이 필요하고, 스플랫은 필요 없습니다. 스플랫은 촬영 당시의 빛이 색에 "구워져" 있습니다. GLB를 위한 조명은 scene.js에 있습니다.

5. 2단계: GLB 모델을 스플랫과 함께 띄우기

연구 질문 중 하나가 "일반 GLB 모델이 가우시안 스플랫과 한 장면에 공존할 수 있는가?"입니다. src/models.js 모듈을 만들어 성의 GLB 메시 모델(castle_of_loarre.glb)을 불러오고, 스플랫(나비 샘플)과 같은 장면에 배치했습니다. 결과: 공존합니다. 나비 스플랫이 성 메시 앞에 자연스럽게 떠 있고, 깊이(depth)가 공유되어 서로 올바르게 가려집니다.

비교 연구용 토글 키: M 키는 GLB 메시를, N 키는 스플랫을 켜고 끕니다.

6. 실전 디버깅 기록: "스플랫이 안 나와요"

처음 실행했을 때 HUD에 "Splat: failed"가 떴습니다. 원인은 하나가 아니라 넷이었고, 하나씩 벗겨낸 과정 자체가 좋은 공부가 되어 기록합니다.

원인 1 — 그 PLY는 스플랫이 아니었다

castle_of_loarre.ply의 파일 첫머리를 열어 보면(head 명령) 이렇게 시작합니다:

element vertex 666404
property float x / y / z
element face 732376   ← 삼각형 face 데이터!

face(삼각형)가 있는 PLY는 일반 3D 메시입니다. 진짜 가우시안 스플랫 PLY는 face가 없고, 대신 각 점마다 불투명도(opacity), 크기(scale), 회전(rot), 구면조화(f_dc/f_rest) 같은 속성이 있습니다. 즉 이 파일은 GLB와 같은 사진측량 메시를 PLY로 내보낸 것일 뿐, Spark가 그릴 수 있는 데이터가 아니었습니다. 확장자가 .ply라고 다 스플랫이 아닙니다.

해결: 진짜 3DGS 파일이 생길 때까지 Spark 공식 샘플(https://sparkjs.dev/assets/splats/butterfly.spz, 나비 177,132개 스플랫)을 원격 URL로 로드하도록 했습니다. 자기만의 스플랫을 얻으려면 Polycam / Luma / Scaniverse 같은 앱으로 촬영해 .ply/.spz로 내보내서 public/splats/에 넣고 CONFIG의 url만 바꾸면 됩니다.

원인 2 — 성 안에서 태어난 플레이어

GLB 성 모델의 실제 크기를 재어 보니 약 220 × 75 × 182 미터였습니다(파일 안의 노드 행렬이 원본 좌표를 크게 확대). 시작 위치 [0, 0, 5]는 성 지형 내부였고, 그래서 첫 실행 때 바위 속 같은 화면이 보였던 것입니다. 시작 위치를 성이 한눈에 보이는 [0, 25, 120]으로 옮기고, 넓은 장면에 맞게 이동 속도도 올렸습니다.

원인 3 — LOD를 켜면 안 보인다

lod: true로 하면 LOD 트리는 정상적으로 만들어지는데(213,741개) 화면에는 아무것도 그려지지 않는 현상을 확인했습니다(spark 2.1.0 + vite 개발 서버). lod: false면 같은 파일이 완벽하게 그려집니다. 일단 데스크톱 개발 단계에서는 LOD를 끄고, LOD가 정말 중요해지는 Quest 단계(6단계)에서 재조사하기로 했습니다. 연구 노트에 남길 만한 발견입니다.

원인 4 — 카메라가 안 움직이면 스플랫이 영영 안 나타나는 버그

가장 흥미로운 문제였습니다. 로딩은 성공했는데(스플랫 개수 표시됨) 화면에 안 보이고, 콘솔에서 위치를 아주 살짝만 바꿔 주면 즉시 나타났습니다. Spark 소스 코드를 읽어 보니: 스플랫 정렬 작업이 진행 중일 때 도착한 갱신 요청은 버려지고 재시도되지 않으며, 카메라 시점이 바뀌어야 다시 갱신됩니다. 마우스를 움직이는 보통 사용자는 절대 못 느끼지만, 카메라가 고정된 상태(자동 테스트 등)에서는 스플랫이 영영 안 보이는 것입니다.

해결(임시): 렌더 루프에서 로딩 완료 후 첫 900프레임 동안 30프레임마다 스플랫을 0.1mm씩 밀어 주는 "splat kick"을 넣었습니다(총 3mm, 눈에 안 보임). 타이머(setInterval) 기반으로 하면 브라우저 탭 절전 때문에 불안정해서, 실제로 렌더링되는 프레임 안에서 실행되도록 했습니다. Spark가 업스트림에서 고치면 제거하면 됩니다.

이번 디버깅에서 배운 것

첫째, 증상 하나에 원인이 여러 개일 수 있습니다. 파일 문제를 고치니 위치 문제가, 위치를 고치니 LOD 문제가, LOD를 끄니 갱신 버그가 차례로 드러났습니다. 둘째, 변인을 하나씩 통제하면(같은 파일을 LOD만 끄고 로드해 보기, 수동으로 위치만 바꿔 보기) 원인이 분리됩니다. 셋째, 문서에 없는 동작은 라이브러리 소스 코드를 직접 읽으면 답이 나옵니다. 넷째, "카메라를 움직이면 고쳐지는" 종류의 버그는 대부분 변경 감지(update detection) 문제입니다.

7. 직접 실행해 보기

터미널을 열고 아래를 입력합니다.

cd ~/Desktop/spark-project
npm run dev

브라우저에서 http://localhost:5173 을 열면 몇 초 안에 성 메시 앞에 나비 스플랫이 보입니다. 화면을 클릭하면 마우스 시점 조작이 시작됩니다.

조작동작
화면 클릭마우스 시점 회전 시작 (Esc로 해제)
W / A / S / D앞 / 왼쪽 / 뒤 / 오른쪽 이동
Q / E아래 / 위로 이동 (공중 이동, 장면 관찰용)

MGLB 메시 켜기/끄기

브라우저 콘솔(F12)에서 __research 객체로 장면·카메라·스플랫을 직접 만져볼 수 있습니다(연구용 디버그 핸들).

8. 다음 단계 로드맵

완료: ① 데스크톱 스플랫 렌더링 → ② 마우스 시점 → ③ WASD 이동 → ④ GLB 모델 공존 → ⑤ WebXR 버튼

남은 단계: ⑥ Quest 실기기 테스트(WebXR은 보안 연결이 필요해서 adb reverse나 HTTPS 설정이 필요합니다 — README 참고. LOD 재조사도 여기서) → ⑦ 아바타 이동(avatar.js는 이미 준비됨) → ⑧ 월드 공간 UI(지금의 HTML HUD는 VR 안에서는 보이지 않기 때문) → ⑨ 네트워킹 / AI 에이전트.
